# 2026-05-20 점심 시간대 전환 guard

> 최근 수요 모양이 명확하게 아래를 가리키는 평일 12시 구간에만 보수적으로 작동하는 noon guard와, UI에 숨긴 lag-shape 진단값을 추가했습니다.

언어: [English](../../en/model-improvements/model-improvement-2026-05-20-midday-transition-features.md) / [日本語](../../ja/model-improvements/model-improvement-2026-05-20-midday-transition-features.md)

---

## 왜 필요했나

2026-05-20 예측에서 차트의 12시 구간이 과하게 높게 유지되는 문제가 있었습니다.

전날 실측과 TEPCO 예측은 모두 12:00-13:00에 일시적인 하락을 보였지만, 우리 모델은 다음 신호를 강하게 보고 수요 수준을 계속 높게 잡았습니다.

- `lag_24h` 같은 같은 시간대 수요 기준선
- 기온과 냉방 부하 피처
- 최근 같은 영업일 타입의 같은 시간대 평균

기존 입력은 "수요 수준"을 설명하는 데는 좋았지만, 11시에서 12시로 넘어갈 때 내려가는지, 12시에서 13시로 다시 올라가는지 같은 시간대 전환 모양을 직접 설명하지 못했습니다.

---

## 변경 내용

추론 전용 context 값 5개를 추가했습니다.

- `lag_24h_hourly_delta`
- `lag_168h_hourly_delta`
- `recent_same_business_type_delta_mean`
- `business_midday_x_lag_24h_delta`
- `business_midday_x_recent_delta_mean`

이 값들은 같은 시간대가 최근에 직전 시간 대비 상승했는지, 하락했는지를 설명합니다.

예를 들어 12:00을 평가할 때 `lag_24h_hourly_delta`는 전날 12:00 수요와 전날 11:00 수요의 차이를 계산합니다. 전날 점심 구간에 수요가 내려갔다면, 파이프라인이 그 모양 context를 볼 수 있습니다.

검증 결과, 이 값들을 LightGBM 학습 피처에 바로 넣으면 전역 hourly delta가 9-10시 같은 무관한 시간대까지 흔드는 부작용이 있었습니다. 그래서 학습 피처로 넣지 않고, LightGBM과 기존 guard 이후 `MiddayTransitionGuard`가 사용하도록 했습니다.

- 영업일에만 작동
- 기본 대상 시간은 12:00
- 최근 같은 영업타입 모양이 의미 있는 하락을 보일 때만 작동
- 예측이 그 전환 context보다 과하게 높을 때만, 상한이 있는 부분 하향 보정 적용

UI에 노출하지 않는 내부 일별 진단 JSON에도 새 lag-shape context를 저장하도록 했습니다. 이후 운영 분석에서 오차 원인이 수준 기준선 문제인지, 시간 전환 모양 문제인지 확인할 수 있습니다.

---

## 기대 효과

운영 파이프라인이 다음 상황을 더 잘 구분할 수 있습니다.

- 더운 날이라 수요 수준은 높아야 하는 경우
- 더운 날이지만 12시 구간은 11시 대비 일시적으로 내려갈 수 있는 경우
- 최근 영업일의 점심 시간대 전환 모양이 달라진 경우

이를 통해 점심 시간대 과상향을 줄이되, 모든 평일 12시를 무조건 낮추는 규칙은 만들지 않는 것이 목표입니다.

---

## 메모

- LightGBM 학습 피처 수는 그대로 유지합니다.
- 저장된 LightGBM interval version도 갱신하지 않았으므로 현재 학습 모델과 호환됩니다.
- 새 context 행은 공개 대시보드가 아니라 UI에 숨겨진 내부 진단 JSON에 포함됩니다.
