# 2026-05-20 午後の気温方向性フィーチャー

> 絶対的な気温だけでなく、気温が上昇中か下降中かをモデルが学習できるように、時間別の気温変化フィーチャーを追加した記録。

## なぜ必要だったか

2026-05-20 は 15時前後までは予測が比較的よく追従していたが、その後の実績需要が、このモデルとTEPCO予測の両方よりも速く低下した。

天気入力には午後ピーク後の気温低下がすでに入っていた。問題は気温データがなかったことではなく、モデルが気温の方向性を直接見にくかったことにある。

- 午前は気温上昇が冷房需要を素早く押し上げることがある。
- 午後は同じ気温でも下降中であれば、オフィスの冷房負荷が速く下がることがある。
- 同じ27度でも、上昇中の27度と下降中の27度では電力需要の反応が異なり得る。

運用予測の観点では、これはヒステリシス型の誤差と見なせる。電力需要は現在の気温水準だけに反応するのではなく、その気温に到達するまでの経路にも影響される。暑い午後には冷房負荷が蓄積するが、業務時間の後半に気温と体感温度が下がり始めると、オフィス冷房や在室人数に由来する需要が、絶対気温だけを見るフィーチャーが想定するより速くほどける場合がある。

既存のフィーチャーセットは、絶対的な暑さ/寒さ、直近平均との差、3日間の熱慣性をすでに表現していた。不足していたのは、直近1~2時間で天気がどちらの方向へ、どれくらい速く動いたかという情報である。この情報がないと、モデルは下降中の午後27度と上昇中の午前27度を過度に似た状態として扱う可能性がある。

## 変更内容

LightGBM入力に短期の気温方向性フィーチャーを追加した。

- `temp_delta_1h`
- `temp_delta_2h`
- `apparent_temp_delta_1h`
- `cooling_delta_1h`

平日15時から18時には、午後後半の相互作用フィーチャーを追加した。

- `business_late_afternoon_x_temp_delta_1h`
- `business_late_afternoon_x_cooling_delta_1h`

これらは需要を強制的に下げるルールベース補正ではない。気温が下がると interaction 値は負になり、上がると正になる。LightGBMが過去需要と一緒にこの関係を学習できるよう、入力だけを提供する。

---

## 期待される効果

モデルが次の状況をより区別しやすくなる。

- 暑くなりながら上昇する平日午前需要
- 暑いが冷え始めている午後需要
- 退勤前後に冷房負荷が速く下がる業務日後半のパターン

既存の実績傾きベースの下落緩和 guard とは役割が異なる。今回のフィーチャーは次の予測でパターンを学習するための入力であり、intraday guard は実績需要がすでに急落し始めた場合だけ運用中の補正として反応する。

## メモ

- 15時から18時に必ず需要を下げるハードコード補正は追加していない。
- これらのフィーチャーは、学習と推論の両方で時間別の予報/観測気温から計算される。
- 内部運用レポートでも気温方向性フィーチャーを確認できるようにした。
- 新しいフィーチャーセットと既存モデルが混ざらないように、LightGBM interval version を上げた。
