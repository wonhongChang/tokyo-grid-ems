# 2026-05-16 営業タイプ遷移lag特徴量

> `lag_24h` が週末/平日の境界をまたぎ、異なる需要レジームを指す問題を減らすための特徴量改善記録です。

言語: [English](../../en/model-improvements/model-improvement-2026-05-16-business-type-lag-features.md) · [한국어](../../ko/model-improvements/model-improvement-2026-05-16-business-type-lag-features.md)

---

## なぜ必要だったか

モデルはすでに対象日の曜日と週末/祝日フラグを持っています。しかし、`lag_24h` が営業日/非営業日の境界をまたぐ場合、依然として危険な基準点になり得ます。

例:

- 土曜昼の `lag_24h` は金曜昼の需要です。
- 月曜昼の `lag_24h` は日曜昼の需要です。

これらの時間帯では、オフィス、商業施設、産業需要の差が大きく出ることがあります。モデルが高い、または低い `lag_24h` だけを見ると、前日の需要を信頼しすぎる可能性があります。

## 予測改善内容

LightGBM特徴量に次の値を追加しました。

- `lag_24h_business_type_mismatch`: 対象日と前日で営業/非営業タイプが異なるか
- `lag_24h_mismatch_x_business_hour`: 同じmismatchを08:00-18:00の日中に集中して示す値
- `recent_same_business_type_mean`: 直近の同じ営業タイプ・同時刻平均

これにより、金曜→土曜、日曜→月曜の `lag_24h` を、平日→平日または週末→週末の `lag_24h` と別に解釈できるようになります。

## 設計上の境界

この変更は土曜予測を手動で下げるルールではありません。LightGBMが過去結果から `lag_24h` をどの程度信頼するか判断できるように文脈を追加します。

`lag_168h` は変更しません。通常は同じ曜日を比較するため、週末予測では自然な基準線である可能性が高いからです。

## 運用メモ

特徴量カラム数が34個から37個に変わったため、LightGBMモデル互換バージョンを上げました。既存の保存モデルはstale扱いとなり、次回のETLまたはintraday実行で再学習されます。
